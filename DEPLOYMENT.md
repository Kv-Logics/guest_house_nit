## Overview

The application runs as three Docker containers managed by `docker compose`:

```
rooms.nitt.edu
│
├── [Reverse Proxy — Nginx/Apache on RHEL host]
│     https://rooms.nitt.edu  ──→  localhost:9006
│
├── Port 9006 (host) ──→  frontend container (Nginx :80)
│                               │ /api/*      → backend container (:5000)
│                               │ /uploads/*  → backend container (:5000)
│                               │ /*          → React SPA (index.html)
│
├── Port 45432 (host) ──→  postgres container (:5432)
│                          (random host port, not publicly exposed)
│
└── Named Docker Volumes
      ├── pgdata   → Postgres data (persists across redeploys)
      └── uploads  → Uploaded files: documents, payments, invoices
```

**No SSL inside Docker.** SSL is terminated by the RHEL host's reverse proxy, which forwards plain HTTP to port `9006`.

## Developer Steps (FileZilla)

> ⚠️ Do these **before** asking the admin to register the runner.

### Step 1 — Fill in `.env.production`

Open the file at the project root and fill in the two required secrets:

```env
DB_PASSWORD=your_strong_random_password_here
JWT_SECRET=paste_64_byte_hex_here
```

Generate the JWT secret locally:
```bash
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"
```

Everything else (domain, CORS, ports) is already configured. For SMTP/email, you can leave it blank and configure it later via the Admin UI.

### Step 2 — Connect to RHEL via FileZilla

| Field | Value |
|-------|-------|
| Host | `rooms.nitt.edu` |
| Port | `21` (FTP) or `22` (SFTP — preferred) |
| Credentials | Your FTP account credentials |

### Step 3 — Create `github-runner/` folder

In FileZilla, navigate to `/var/www/rooms.nitt.edu/` and create a new empty folder named exactly `github-runner`.

### Step 4 — Upload `.env.production`

Upload your filled-in `.env.production` file to `/var/www/rooms.nitt.edu/`.

```
/var/www/rooms.nitt.edu/
├── github-runner/        ← created ✅
└── .env.production       ← uploaded ✅
```

**You are done. Tell the admin the folder is ready.**

## Admin Steps (Terminal — One Time Only)

> These steps are run **once** by a server admin with SSH access. The developer does not need to do this.

### Step 1 — Install GitHub Runner Binaries (if not already done)

```bash
# Create shared runner binary directory (shared across all projects on this server)
sudo mkdir -p /opt/github-runner/bin
cd /opt/github-runner/bin

# Download runner (check https://github.com/actions/runner/releases for latest version)
curl -o actions-runner.tar.gz -L \
  https://github.com/actions/runner/releases/download/v2.316.1/actions-runner-linux-x64-2.316.1.tar.gz

tar xzf ./actions-runner.tar.gz
```

> ℹ️ This only needs to be done **once for the entire server**, not once per project.

### Step 2 — Get the Registration Token from GitHub

1. Go to `https://github.com/Kv-Logics/guest_house_nit`
2. Navigate to **Settings → Actions → Runners → New self-hosted runner**
3. Copy the `--token` value from the displayed command. It expires in **1 hour** — use it immediately.

### Step 3 — Register the Runner for this Project

```bash
# Navigate into the runner binaries
cd /opt/github-runner/bin

# Register — points connection keys to the developer's github-runner folder,
# and sets the working directory to where the code will be deployed
./config.sh \
  --url https://github.com/Kv-Logics/guest_house_nit \
  --token <PASTE_TOKEN_HERE> \
  --name "rooms-nitt-edu-runner" \
  --work /var/www/rooms.nitt.edu/guesthouse \
  --runnergroup Default \
  --labels self-hosted,RHEL \
  --unattended

# Move generated key files to the developer's folder
mv /opt/github-runner/bin/.runner /var/www/rooms.nitt.edu/github-runner/
mv /opt/github-runner/bin/.credentials /var/www/rooms.nitt.edu/github-runner/
```

### Step 4 — Start the Runner as a Background Service

```bash
# Option A: Run in background using nohup (simple)
cd /opt/github-runner/bin
nohup ./run.sh \
  --work /var/www/rooms.nitt.edu/guesthouse \
  > /var/www/rooms.nitt.edu/github-runner/runner.log 2>&1 &

echo "Runner started. PID: $!"
```

```bash
# Option B: Run as a systemd service (recommended for production — survives reboots)
sudo tee /etc/systemd/system/github-runner-rooms.service > /dev/null <<EOF
[Unit]
Description=GitHub Actions Runner — rooms.nitt.edu
After=network.target docker.service
Requires=docker.service

[Service]
Type=simple
User=www-data
WorkingDirectory=/opt/github-runner/bin
ExecStart=/opt/github-runner/bin/run.sh --work /var/www/rooms.nitt.edu/guesthouse
Restart=always
RestartSec=5
StandardOutput=append:/var/www/rooms.nitt.edu/github-runner/runner.log
StandardError=append:/var/www/rooms.nitt.edu/github-runner/runner.log

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable github-runner-rooms
sudo systemctl start github-runner-rooms
sudo systemctl status github-runner-rooms
```

### Step 5 — Verify Runner is Online

Go to **GitHub → Settings → Actions → Runners** and confirm the runner shows as **Idle** (green dot).

## How CI/CD Works After Setup

Every time a developer runs `git push origin main`, the following happens automatically:

```
1. GitHub detects push to `main`
2. GitHub sends job to the self-hosted runner on rooms.nitt.edu
3. Runner executes .github/workflows/deploy.yml:
   a. git pull (latest code → /var/www/rooms.nitt.edu/guesthouse/)
   b. cp /var/www/rooms.nitt.edu/.env.production .env.production
   c. docker compose --env-file .env.production up --build -d
   d. docker image prune -f
4. New containers are live at http://rooms.nitt.edu:9006
```

The deploy workflow file is at [`.github/workflows/deploy.yml`](.github/workflows/deploy.yml).

## Environment Variables Reference

All configuration lives in a single `.env.production` file (never committed to git).

```env
# ── Database ──────────────────────────────────────────────────────
DB_USER=postgres
DB_HOST=db                         # Docker service name — do not change
DB_NAME=guesthouse_db
DB_PASSWORD=<FILL THIS IN>         # Strong random password
DB_PORT=5432                       # Internal Docker port — do not change

# ── Backend ───────────────────────────────────────────────────────
PORT=5000
NODE_ENV=production
JWT_SECRET=<FILL THIS IN>          # 64-byte hex (see generation command above)

# ── CORS ──────────────────────────────────────────────────────────
# Both http (direct port access) and https (via reverse proxy) are allowed
CORS_ORIGIN=http://rooms.nitt.edu:9006,https://rooms.nitt.edu,http://localhost:9006

# ── Cookie Security ───────────────────────────────────────────────
# false = works over plain HTTP (before SSL reverse proxy)
# true  = set this after SSL reverse proxy is active
COOKIE_SECURE=false

# ── SMTP / Email ──────────────────────────────────────────────────
# Optional — can be configured in Admin UI instead
SMTP_HOST=
SMTP_PORT=587
SMTP_USERNAME=
SMTP_PASSWORD=
SMTP_SECURE=tls
```

## First Deployment — Database Init

On the **very first** `docker compose up`, the Postgres container is empty. The project's `backend/schema.sql` is mounted to `/docker-entrypoint-initdb.d/` and runs automatically — creating all tables, sequences, and seed data.

This only happens once. Subsequent deployments preserve all data via the `pgdata` named Docker volume.

> ⚠️ If you ever need to **reset the database** (e.g., in testing):
> ```bash
> docker compose down -v   # WARNING: destroys all data
> docker compose --env-file .env.production up --build -d
> ```

## Reverse Proxy Setup (Nginx on RHEL)

The admin configures the RHEL host's Nginx to forward traffic from the domain to Docker's port `9006`.

```nginx
# /etc/nginx/conf.d/rooms-nitt-edu.conf

server {
    listen 80;
    server_name rooms.nitt.edu;

    location / {
        proxy_pass         http://localhost:9006;
        proxy_http_version 1.1;
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_read_timeout 120s;
        client_max_body_size 15M;
    }
}
```

```bash
sudo nginx -t && sudo systemctl reload nginx
```

## Switching to HTTPS

Once the RHEL admin adds SSL (via Let's Encrypt / institutional certificate):

**1. Update `.env.production`:**
```env
COOKIE_SECURE=true
```

**2. Re-upload the file via FileZilla** to `/var/www/rooms.nitt.edu/.env.production`

**3. Trigger a redeploy:**
```bash
git commit --allow-empty -m "chore: enable COOKIE_SECURE for HTTPS"
git push origin main
```

CORS is already configured for `https://rooms.nitt.edu` — no other changes needed.

## Useful Docker Commands

Run these on the RHEL server via SSH (admin) or in a terminal session:

```bash
# View running containers
docker compose ps

# View live logs from all containers
docker compose --env-file .env.production logs -f

# View logs from backend only
docker compose --env-file .env.production logs -f backend

# Restart a single container without rebuild
docker compose --env-file .env.production restart backend

# Full rebuild and restart
docker compose --env-file .env.production up --build -d

# Stop everything (data is preserved in volumes)
docker compose down

# Check disk usage
docker system df
```

## Troubleshooting

### App not accessible at `http://rooms.nitt.edu:9006`
```bash
# Check all containers are running
docker compose ps

# Check frontend container logs
docker compose logs frontend

# Check if port 9006 is listening
ss -tlnp | grep 9006
```

### Login not working (cookie issues)
- If accessing via HTTP: ensure `COOKIE_SECURE=false` in `.env.production`
- If accessing via HTTPS: ensure `COOKIE_SECURE=true` in `.env.production`
- After changing, re-upload `.env.production` and push to trigger redeploy

### Backend crashing on startup
```bash
docker compose logs backend
# Most common cause: DB not ready yet. 
# The healthcheck retries 5 times with 10s interval.
# If it persists, check DB_PASSWORD matches between .env.production and Postgres volume.
```

### Database connection refused
```bash
# Check postgres container is healthy
docker compose ps db
# Should show: "healthy"

# If volume data is corrupt, reset (WARNING: destroys all data)
docker compose down -v
docker compose --env-file .env.production up --build -d
```

### PDF generation failing (Puppeteer)
The backend Dockerfile installs all required Chromium dependencies. If PDFs still fail:
```bash
docker compose logs backend | grep -i puppeteer
# Check for missing library errors and add to backend/Dockerfile
```

### Uploads not persisting across redeploys
Uploads are stored in a named Docker volume (`uploads`). As long as you use `docker compose up` (not `docker compose down -v`), uploads persist. Verify:
```bash
docker volume ls | grep uploads
docker volume inspect guesthouse_uploads
```

### GitHub runner shows Offline
```bash
# Check runner service status
sudo systemctl status github-runner-rooms

# View runner logs
tail -f /var/www/rooms.nitt.edu/github-runner/runner.log

# Restart runner
sudo systemctl restart github-runner-rooms
```
